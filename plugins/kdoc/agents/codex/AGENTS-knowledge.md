## Knowledge Documentation (kdoc)

This project uses **kdoc** for structured Knowledge documentation. The following rules and templates apply to all Codex agents working on this project.

---

### Knowledge Structure

```
Knowledge/
├── ADR/                    # Architecture Decision Records (ADR-NNNN-*.md)
├── TLDR/                   # Functional requirements per module
│   ├── admin/
│   ├── shop/
│   └── shared/
├── Roadmap/
│   ├── phases/             # phase-N.md + phase-N/M.md for sub-phases
│   └── generated/          # Auto-generated dashboard (do not edit manually)
├── Design/                 # Page/screen/flow specs by pack
│   ├── nextjs/
│   └── swift-ios/
├── Guides/                 # Operational guides
├── AgentMemory/            # Cross-session AI memory
│   ├── MEMORY.md           # Top-level index
│   ├── gotchas-*.md        # Non-obvious facts
│   └── patterns-*.md       # Reusable conventions
├── runbooks/
│   └── threat-models/      # STRIDE threat models
├── Templates/              # Document templates by pack
├── governance-health.md    # Generated by knowledge-auditor agent
└── INDEX.md                # Generated by build_index.py
```

---

### Rules for Codex Agents

**Sync rule:** When making functional code changes, update the corresponding TLDR in `Knowledge/TLDR/`. ADR changes require updating the relevant ADR file.

**ADR policy:** Create an ADR whenever a decision affects architecture, data, integrations, or global standards. File naming: `ADR-{NNNN}-{kebab-title}.md`. Required frontmatter: `id`, `title`, `date`, `status`.

**Gap tag rules:**
- `has-open-questions` — TLDR has unresolved open questions; remove when all questions are struck through
- `missing-test-scenarios` — Test Scenarios table is empty; remove when at least one row is added
- `missing-acceptance-criteria` — Acceptance Criteria is empty; remove when content is added
- `blocked-by-decision` — blocked pending an ADR resolution

**Wikilink rules:** Use `[[Filename]]` (no extension) for cross-references. All wikilinks must resolve to existing files in the Knowledge directory.

**Memory rules:** Save operational facts to `Knowledge/AgentMemory/`. Gotchas go in `gotchas-<topic>.md`, patterns in `patterns-<topic>.md`. Never store secrets or credentials.

**Read-only areas:** `Knowledge/Roadmap/generated/` — generated by roadmap-builder agent. Do not edit manually.

### Multi-Stream Knowledge Audit Templates

Use these stream definitions to run parallel Codex Knowledge audits.

---

#### Stream 1 — ADR Audit

**Goal:** Validate all ADRs for numbering integrity, frontmatter completeness, cross-references, and supersession chain integrity.

**Scope:** `Knowledge/ADR/` — read-only.

**Steps:**
1. Glob `Knowledge/ADR/ADR-*.md` and sort by filename.
2. For each ADR:
   - Check filename matches `ADR-{NNNN}-{slug}.md`.
   - Check frontmatter has: `id`, `title`, `date`, `status`.
   - Check `id` in frontmatter matches NNNN in filename.
   - Check `status` is one of: `proposed`, `accepted`, `rejected`, `superseded`.
   - If `supersedes: ADR-Y` exists, verify ADR-Y exists.
   - Check `[[ADR-NNNN]]` wikilinks resolve to existing files.
3. Extract all NNNNs; report duplicates as FAIL; report gaps as INFO.
4. Output structured report: Total ADRs, status breakdown, failures, warnings, info.

**Output file:** None (read-only stream).

---

#### Stream 2 — TLDR Sync

**Goal:** Validate all TLDRs for gap tag consistency, status lifecycle correctness, and wikilink integrity.

**Scope:** `Knowledge/TLDR/` — read-only.

**Steps:**
1. Glob `Knowledge/TLDR/**/*.md` (excluding `README.md`).
2. For each TLDR:
   - Check frontmatter has: `area`, `status`. Should have: `module`.
   - Check `status` is one of: `draft`, `ready`, `in-progress`, `done`, `deprecated`.
   - Gap tag consistency:
     - `has-open-questions` present → Open Questions section must have non-struck-through items.
     - `missing-test-scenarios` present → Test Scenarios table must be empty or absent.
     - `missing-acceptance-criteria` present → Acceptance Criteria must be empty or absent.
     - Section filled but tag present → WARN (stale tag).
     - Section empty but tag absent → WARN (missing tag).
   - Status alignment: `done` + `has-open-questions` → WARN. `done` + empty Test Scenarios → WARN.
   - Check `[[TARGET]]` wikilinks resolve.
3. Best-effort module coverage: compare module names against TLDR filenames; report gaps as INFO.
4. Output structured report: Total TLDRs, status breakdown, failures, warnings, info.

**Output file:** None (read-only stream).

---

#### Stream 3 — Roadmap Dashboard

**Goal:** Read all roadmap phase files, cross-reference TLDRs and evidence, generate an updated dashboard.

**Scope:** `Knowledge/Roadmap/` — reads phases, writes dashboard only.

**Steps:**
1. Glob `Knowledge/Roadmap/phases/phase-*.md` (top-level phases).
2. Glob `Knowledge/Roadmap/phases/phase-*/*.md` (sub-phases).
3. For each phase: extract `phase`, `title`, `status`, `startDate`, `endDate`, Sub-Phases table.
4. Cross-reference: TLDRs referenced in tasks must exist and have matching status. `complete` phase + `draft` TLDR → WARN.
5. Evidence check: for `complete` sub-phases, listed Evidence Files must exist.
6. Write `Knowledge/Roadmap/generated/dashboard.md`:
   - Phase overview table: phase, title, status, sub-phase count, completion %.
   - Per-phase detail with sub-phases table.
   - Health section: mismatches and coverage.
7. Output summary: total phases, completion %, mismatches count.

**Output file:** `Knowledge/Roadmap/generated/dashboard.md` (only file this stream writes).
